# Workflow Dispatch Telemetry

Workflow telemetry records slash-command dispatch facts mechanically. It does not ask workflow prompts or plans to narrate task, validation, review, or archive events.

## Ownership

`pi/lib/workflow-telemetry.ts` owns the writer. `pi/scripts/workflow-eval-query.py` reads the resulting JSONL for command and event counts. Runtime files remain local under `~/.pi/workflow-telemetry/` and are not committed.

Orchestration telemetry separately owns delegation topology, usage, cost, output handling, and run status. The two streams are not joined by inferred timestamps or names.

## Recorded files

```text
~/.pi/workflow-telemetry/
  episodes.jsonl
  {episode_id}/
    events.jsonl
```

Each command dispatch appends one episode and one event.

Episode fields:

| Field | Meaning |
| --- | --- |
| `schema_version` | Record schema version, currently `1`. |
| `episode_id` | Timestamped command and artifact slug generated by runtime code. |
| `command` | Registered workflow command name. |
| `artifact_path` | Plan path parsed from command arguments, when present. |
| `repo_root` | Runtime working directory. |
| `started_at` | ISO-8601 dispatch timestamp. |
| `status` | `started`; dispatch telemetry does not infer completion. |
| `redaction_status` | Whether sensitive output was recorded; dispatch records contain no command output. |

Dispatch event fields:

| Field | Meaning |
| --- | --- |
| `schema_version` | Record schema version, currently `1`. |
| `episode_id` | Parent episode ID. |
| `event_id` | `dispatch-001`. |
| `phase_id` | `dispatch`. |
| `event_type` | `command`. |
| `command_line` | Slash command and arguments. |
| `status` | `recorded`. |
| `evidence` | Runtime dispatch fact. |
| `created_at` | ISO-8601 dispatch timestamp. |

## Non-goals

- Inferring implementation completion, validation results, blockers, review yield, manual decisions, or archive state.
- Reconstructing detailed execution from model-authored schema-shaped prose.
- Writing plans, review artifacts, DuckDB files, or raw command output.
- Replacing plan checklists and bounded evidence summaries.

## Query

From the repository root:

```bash
python pi/scripts/workflow-eval-query.py
```

Use a separate telemetry root for isolated validation:

```bash
python pi/scripts/workflow-eval-query.py --telemetry-dir .tmp/workflow-telemetry --no-duckdb
```

The helper reads episode and event JSONL and reports command and event counts. DuckDB is optional and remains a rebuildable query engine rather than a write target.
