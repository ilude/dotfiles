## Summary

The two-day sprint lands a commit extension, an observability timing layer, dual
lockfiles in the test tree, and three parallel commit implementations. The
module boundary between `pi/lib/` and `pi/extensions/` is clean and intentional.
Most findings are scoped to a missing error-boundary in the new commit tools, the
dual-lockfile situation, observability adoption coverage, and a documentation

## Findings

### BLOCKER

#### 1. `commit_stage` and `commit_create` execute handlers expose raw throws to the Pi framework

**Severity:** Blocker  
**File:** `pi/extensions/commit.ts:62-77`

**Issue:** `stagePaths()` and `createCommit()` both throw `Error` on failure (confirmed in
`pi/lib/commit/create.ts:15,23-33`, `pi/lib/commit/stage.ts`). The `execute()` handlers in
`commit.ts` call them with no `try/catch`, so any failure propagates as an uncaught async
exception rather than a `formatToolError` return. The `pi/extensions/README.md` states
that tool `execute()` handlers MUST return errors via `formatToolError` so the UI and
downstream LLM see consistent error content. Every other execute handler in the codebase
that calls fallible lib functions (e.g. `permissions.ts:128,173,197`) wraps in `try/catch`.

The path is definitely reachable: a wrong confirmation token, a `git add` failure, or a
staged-set mismatch all throw inside `createCommit()`.

**Suggested fix:** Wrap both mutating handlers in `try/catch` and return
`formatToolError(err.message)` on catch, following the pattern in `permissions.ts` and
per the README contract:

```ts
execute: async (_toolCallId, params, _signal, _onUpdate, ctx) => {
  try {
    const result = stagePaths(...);
    return toolResult(`Staged ${result.staged.length} path(s).`, result);
  } catch (err) {
    return formatToolError(err instanceof Error ? err.message : String(err));
  }
},
```

### FOLLOW-UP

#### 2. Dual lockfiles in `pi/tests/` -- `bun.lock` and `package-lock.json` coexist

**Severity:** Medium follow-up  
**File:** `pi/tests/bun.lock`, `pi/tests/package-lock.json`

`pi/tests/package.json` runs tests via `bun vitest`. The directory contains both a
`bun.lock` (lockfileVersion 1) and a `package-lock.json` (lockfileVersion 3, generated by
npm). The npm lockfile represents a different resolved dependency graph; `bun` will ignore
it but a CI environment that runs `npm ci` would install different versions. The project
CLAUDE.md and README.md both say "never npm". The `package-lock.json` appears to be an
accidental artefact of a one-off `npm install` invocation.

**Suggested fix:** Delete `pi/tests/package-lock.json` and add `pi/tests/package-lock.json`
to `.gitignore` to prevent recurrence.

#### 3. Observability adoption is partial -- the new commit extension has no timing instrumentation

**Severity:** Low follow-up  
**File:** `pi/extensions/commit.ts` (no timing); compare with `pi/extensions/workflow-commands.ts:799` and `pi/extensions/subagent/index.ts:395`

`withTimingSpan` / `TimingSpan` are used in two of the recently-touched extensions
(`workflow-commands.ts` wraps `/review-it`; `subagent/index.ts` wraps the full run).
The new `commit_stage` and `commit_create` tools -- the most latency-sensitive operations
in the new flow -- carry no spans. This is not a bug, but it means the observability layer
that was built specifically for this sprint produces no data for the commit path.

**Suggested fix:** Wrap `commit_stage` and `commit_create` execute handlers with
`withTimingSpan` (category `"tool"`, name e.g. `"commit.stage"` / `"commit.create"`).
This pairs naturally with fixing finding #1 above since both changes touch the same
execute blocks.

#### 4. Commit implementation is intentionally three-layer but the contract doc is stale

**Severity:** Low follow-up  
**File:** `docs/commit-helper-contract.md:3`

`docs/commit-helper-contract.md` says: "Pi now owns the canonical commit workflow through
the TypeScript Pi commit extension (`pi/extensions/workflow-commands.ts` plus
`pi/extensions/commit.ts`)." This is correct as of this sprint. However the
`claude/agents/committer.md` agent still exists as a third consumer and is not mentioned
in the contract doc. The three-layer split is intentional per the archived spec:
Python helper = deterministic planning for non-Pi clients; workflow-commands = `/commit`
slash-command (LLM-driven grouping); `pi/lib/commit/*` + `commit.ts` = native Pi tools.
The layering is sound, but the contract doc should acknowledge all three layers and their
migration status to prevent future confusion about which to extend.

**Suggested fix:** Add a one-paragraph "Commit surface inventory" section to
`docs/commit-helper-contract.md` naming all three surfaces and their intended consumers.

### QUESTIONS

#### 5. `pi/multi-team/expertise/` and `pi/index/` runtime files in session git status

**Severity:** Question  
**Area:** `.gitignore` vs tracked content

The git status snapshot at session start showed several files under
`pi/multi-team/expertise/` and `pi/index/` as modified (`M`). Investigation confirms:
- Both directories are listed in `.gitignore` (lines 104, 105).
- `git ls-files` returns no tracked files in either directory.
- `git status --ignored` shows `!!` (correctly ignored) for both.

The `M` lines in the snapshot appear to reflect unstaged changes to files that are
ignored-but-present on disk (perhaps from a prior `git add -f` that was later reverted,
or a `.gitignore` rule that was added after the files were already on disk in an untracked
state). No tracked runtime-state file was found in the diff.

**Question for the team:** Were any of these files ever force-committed and later removed
from tracking? If so, a `git rm --cached` sweep may be needed. If the session-start
snapshot is simply showing pre-existing untracked-but-present files, no action is needed.

## Verified Safe

- **Module boundary (`pi/lib/` vs `pi/extensions/`):** Extensions import from `lib`
  (expected). `lib` files do not import from `extensions` (the one grep hit was a comment
  in `lib/operator-state.ts`, not an import statement). Boundary is clean.
- **Extension registration:** `commit.ts` exports a valid `default function(pi)` factory
  and is auto-discovered. No registration gap. `commit_plan`, `commit_validate_message`,
  `commit_stage`, `commit_create` are all registered. The test `commit-extension.test.ts`
  asserts single ownership of `/commit` in `workflow-commands.ts` and confirms no leakage
  of `registerCommitTools` into `workflow-commands`.
- **`pi/extensions/package.json` dep drift:** The `pnpm-lock.yaml` resolves all five
  declared dependencies. No unpinned or phantom dependencies found. Versions are consistent
  with `^0.68.1` specifiers locked to `0.68.1`.
- **`TimingSpan` lifecycle in `subagent/index.ts`:** All exit paths (abort, success,
  non-zero exit, unexpected throw) call `timingSpan.finish()` exactly once via the
  `timingFinished` guard. No span leak.
