# V1.1 Amendment: Warp-Inspired UX Layer

## Purpose

This amendment adds a lightweight workflow and usability layer on top of the v1 cockpit without changing the core architecture.

It is intended to close the biggest practical gaps between the current plan and a more productized terminal experience like Warp:
- discoverability
- first-run guidance
- health checks and recovery
- easier session re-entry

This amendment does **not** attempt to add:
- command-block rendering
- a custom terminal UI framework
- multi-agent orchestration
- Pi RPC management
- global Zellij plugin work

## Design Principle

Keep the stack modular:
- PowerShell remains the workflow glue
- Zellij remains the workspace/layout manager
- Micro remains the editor
- Yazi remains the file manager
- Pi remains the agent

The UX improvements should come from a thin repo-managed command layer, not from replacing the stack.

## New User-Facing Commands

### `cockpit`
A discoverable action menu for common cockpit actions.

Suggested actions:
- Open cockpit for project
- Reattach/open cockpit here
- Open Yazi here
- Open file in Micro
- Launch Pi here
- Show help
- Run health check
- Show recent projects

### `cockpit-check`
A health check command that validates the expected v1 toolchain and prints actionable fixes.

Checks should cover at least:
- PowerShell 7 session
- `zellij`
- `micro`
- `yazi`
- `pi`
- `fzf`
- `fd`
- `rg`
- `bat`
- expected repo-owned layout path

### `cockpit-help`
A human-readable help command for first-run and recovery.

It should explain:
- what the cockpit is
- what each pane does
- `Ctrl+g` lock/unlock behavior
- what `zproj` does
- what to do when Pi is missing
- what to do when a new PowerShell session is required
- current v1 limitations

### Smarter `zproj`
Enhance `zproj` so it prefers attaching to an existing session for the selected project and only creates a new session when needed.

Desired flow:

```text
pick project/worktree
  → derive session name
  → if session exists, attach
  → else create session with dev layout
```

## Scope Boundaries

### In scope
- PowerShell workflow commands
- helper-driven discoverability
- health checks and recovery text
- recent-project recall if it can be implemented simply
- smarter session resume behavior
- right-pane help/dashboard improvements if they remain static and low-risk

### Out of scope
- replacing Zellij
- replacing PowerShell helpers with a custom TUI app
- structured command blocks
- persistent activity/event database
- dynamic agent roster UI
- cross-platform implementation beyond what falls out naturally from existing patterns

## Detailed Task Notes

### A1: Add `cockpit` action menu and shared dispatch helpers
- Description: Add a discoverable command menu, likely using `fzf`, that exposes the primary cockpit actions without requiring the user to memorize helper names.
- Files:
  - `powershell/profile.ps1` or a sourced repo-owned PowerShell module
- Acceptance Criteria:
  1. [ ] `cockpit` is defined and launches a menu of at least 5 useful actions.
     - Verify: `pwsh -NoProfile -Command ". ./powershell/profile.ps1; Get-Command cockpit | Select-Object Name"`
     - Pass: `cockpit` is defined and callable
     - Fail: command missing or profile fails to load
  2. [ ] The menu includes direct access to `zproj`, `y`, `ff`, `cockpit-help`, and `cockpit-check`.
     - Verify: `rg -n "cockpit-help|cockpit-check|zproj|ff|\by\b" powershell/profile.ps1`
     - Pass: menu dispatch clearly includes the required actions
     - Fail: the action menu is incomplete or hardcodes broken references
  3. [ ] The command degrades clearly when `fzf` is missing.
     - Verify: inspect command logic in profile/module
     - Pass: user gets a readable error or fallback guidance instead of silent failure
     - Fail: menu assumes `fzf` exists with no guidance

### A2: Add `cockpit-check` health check and actionable recovery output
- Description: Add a toolchain and environment health check for the cockpit with readable OK/missing output and concrete fix hints.
- Files:
  - `powershell/profile.ps1` or a sourced repo-owned PowerShell module
- Acceptance Criteria:
  1. [ ] `cockpit-check` validates required binaries and key environment assumptions.
     - Verify: `pwsh -NoProfile -Command ". ./powershell/profile.ps1; Get-Command cockpit-check | Select-Object Name"`
     - Pass: `cockpit-check` exists
     - Fail: command missing or profile fails to load
  2. [ ] Missing tools produce specific remediation steps.
     - Verify: `rg -n "npm install -g @mariozechner/pi-coding-agent|winget|missing|not found|PowerShell 7" powershell/profile.ps1`
     - Pass: output logic includes concrete fixes for at least Pi and missing core tools
     - Fail: health check only reports missing state without remediation
  3. [ ] The health check covers the repo-owned layout/config path expected by `zproj`.
     - Verify: `rg -n "config/zellij|layouts/dev\.kdl" powershell/profile.ps1`
     - Pass: the check validates that expected repo assets exist
     - Fail: health check ignores the repo-owned launch assets

### A3: Add `cockpit-help` and first-run guidance content
- Description: Add a concise help command plus any supporting text that makes first-run behavior and current limitations explicit.
- Files:
  - `powershell/profile.ps1`
  - optionally `config/zellij/layouts/dev.kdl` if pane text needs alignment
- Acceptance Criteria:
  1. [ ] `cockpit-help` explains pane roles, `Ctrl+g`, shell expectations, and current v1 limitations.
     - Verify: `rg -n "Ctrl\+g|PowerShell 7|v1|static|Pi|Yazi|Micro|Zellij" powershell/profile.ps1 config/zellij/layouts/dev.kdl`
     - Pass: the help text covers first-run and current limitations clearly
     - Fail: important behavior still relies on tribal knowledge
  2. [ ] Help text is consistent with the actual v1 implementation.
     - Verify: manual review against `.specs/zellij-windows-cockpit-v1/plan.md`
     - Pass: help does not promise features that v1 does not implement
     - Fail: help text drifts into deferred ideas like dynamic agent switching

### A4: Upgrade `zproj` to attach-or-create behavior
- Description: Improve `zproj` so re-entry is smoother: attach to an existing session for the selected project when present, otherwise create one using the v1 launch contract.
- Files:
  - `powershell/profile.ps1`
- Acceptance Criteria:
  1. [ ] `zproj` checks for an existing Zellij session before creating a new one.
     - Verify: `rg -n "attach|session|list-sessions|zellij" powershell/profile.ps1`
     - Pass: logic clearly distinguishes attach vs create
     - Fail: command always creates a new session or leaves session behavior ambiguous
  2. [ ] Session naming remains deterministic for the same project/worktree.
     - Verify: inspect naming logic in `zproj`
     - Pass: identical project selection resolves to the same normalized session name
     - Fail: session names drift across launches
  3. [ ] The attach-or-create flow still uses the repo-owned dev layout when creating a new session.
     - Verify: `rg -n "layouts/dev\.kdl|--layout dev|config/zellij" powershell/profile.ps1`
     - Pass: new sessions still launch the intended cockpit layout
     - Fail: session resume logic breaks the original cockpit launch contract

### A5: Validate the v1.1 workflow layer end-to-end
- Description: Validate that the new UX layer improves discoverability and recovery without destabilizing v1.
- Files: none
- Acceptance Criteria:
  1. [ ] PowerShell loads with `cockpit`, `cockpit-check`, `cockpit-help`, and upgraded `zproj` available.
     - Verify: `pwsh -NoProfile -Command ". ./powershell/profile.ps1; Get-Command cockpit,cockpit-check,cockpit-help,zproj | Select-Object Name"`
     - Pass: all commands are available
     - Fail: profile load errors or missing commands
  2. [ ] A first-time user can discover how to launch the cockpit and diagnose missing tools without reading the spec.
     - Verify: manual review of command/help/check output
     - Pass: help/check outputs provide enough guidance to proceed
     - Fail: first-run still depends on memorized commands or hidden assumptions
  3. [ ] Existing v1 launch behavior still works.
     - Verify: run v1 validation steps from `.specs/zellij-windows-cockpit-v1/plan.md`
     - Pass: v1 remains intact while the workflow layer improves usability
     - Fail: new workflow helpers regress or replace core v1 behavior incorrectly

## Recommended Execution Order

```text
A1, A2, A3 (parallel) → A4 → A5
```

Reasoning:
- menu, health check, and help can be built independently once v1 exists
- `zproj` attach-or-create should incorporate the help/check assumptions but remain a separate integration step
- final validation should assess the combined workflow

## Notes on Effort vs Value

### Highest value, lowest risk
- `cockpit-check`
- `cockpit-help`
- smarter `zproj`

### Highest UX improvement
- `cockpit` action menu

### Things to avoid in v1.1
- do not build a custom TUI app
- do not build a persistent state store unless the simple workflow layer proves insufficient
- do not let help/dashboard text drift into future multi-agent semantics

## Success Criteria

V1.1 succeeds if:
1. users can discover core actions without memorizing helper names
2. missing tools produce clear fixes instead of confusing failures
3. re-entering an existing cockpit session is easier and more deterministic
4. the stack remains modular and repo-managed rather than turning into a new monolithic app
